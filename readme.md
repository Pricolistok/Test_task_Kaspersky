# Wordforms Report

Реализация тестового задания: API для анализа текстового файла и построения статистики по словоформам.

## Что делает сервис

- принимает `.txt` файл через API  
- считает частоту словоформ
  - общее количество по документу  
  - количество в каждой строке  
- сохраняет результат в `.xlsx`  
- даёт возможность:
  - проверить статус задачи  
  - скачать результат  

---

## Как это работает

### 1. Загрузка файла

POST /api/v1/public/report/export

Файл:
- должен быть `.txt`
- кодировка — UTF-8

Ответ:

{
  "task_id": "..."
}

---

### 2. Проверка статуса

GET /api/v1/public/report/get_status_task/{task_id}

Ответ:

{
  "status": "processing",
  "detail": "Analyzing wordforms",
  "last_update": "2026-..."
}

Статусы:

- queued — задача в очереди  
- processing — выполняется  
- done — готово  
- failed — ошибка  

---

### 3. Скачать результат

GET /api/v1/public/report/get_file/{task_id}

Возвращает `.xlsx` файл.

---

## Почему так сделано

Основная проблема - большие файлы и несколько пользователей.

Если несколько пользователей одновременно запустят на обработку большие файлы, то
сервер просто подвиснет, поэтому я реализовал очередь, благодаря которой файл кладется в неё,
пользователю возвращается номер задачи, по которому он может отслеживать её статус. Тем самым
сервер не блокируется от наплыва файлов, а постепенно обрабатывает все заявки.

Поэтому сделано через очередь и worker.

### Очередь задач

- используется `queue.Queue(maxsize=25)`  
- ограничивает количество задач в памяти  
- если очередь переполнена, то возвращается ошибка `503`  

### Worker (фоновой поток)

- запускается при старте приложения  
- постоянно слушает очередь  
- берёт задачи и обрабатывает их  

---

## Как считается статистика

- файл читается построчно  
- слова нормализуются через pymorphy3  
- данные хранятся в виде словаря, в котором ключ — лемма, а значение — класс, который
хранит в себе общую статистику по файлу и построчную статистику:

lemma: { total: int, lines: {line_id: count} }

Построчная статистика хранится в разреженном виде, это
сделано, чтобы не забивать память нулями и экономить память.

---

## Выгрузка в Excel

Выгрузка происходит построчно, что позволяет сэкономить память, так как в моменте мы 
храним только одну строку.

---

## Ограничения решения

- очередь и статусы хранятся в памяти процесса  
- при перезапуске сервера состояние задач теряется  

Какие улучшения возможны:
- вынести очередь в Redis  
- хранить статусы в БД  
- хранить леммы в БД, так как это со временем позволило бы накопить 
большую библиотеку слов и обрабатывать файлы быстрее

## Итог

Основная идея решения:
- не блокировать API тяжёлой работой  
- контролировать нагрузку через очередь  
- обрабатывать файлы потоково  
- минимизировать использование памяти  


## Тестирование

Для проверки работы программы была написана программа, которая генерирует большие текстовые файлы.

После этого был запущен сервер, и несколько раз подряд отправлялся большой файл через API.  
Это позволило создать нагрузку на очередь: задачи не выполнялись мгновенно, а обрабатывались постепенно.

Таким образом удалось:
- проверить корректность работы очереди  
- убедиться, что сервер не блокируется при нагрузке  
- отследить все этапы выполнения задач
